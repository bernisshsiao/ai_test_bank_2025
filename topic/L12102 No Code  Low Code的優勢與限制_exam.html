L12102 No Code/Low Code的優勢與限制
出題方向
9
No-Code/Low-Code 概念與應用
#1
★★★★★
下列何者最能表達 No Code/Low Code 平台的主要特色？
A
需要撰寫大量程式碼
B
運用模板或視覺化介面快速建立應用程式
C
僅供專業開發人員使用
D
只能製作靜態網站
答案解析
No Code/Low Code 平台的核心價值在於降低應用程式開發的門檻，讓使用者（包括非專業開發人員，即公民開發者 Citizen Developer）能夠透過視覺化的拖放介面、預設的模板和元件，快速地建構和部署應用程式，而無需或只需少量編寫程式碼。這大大縮短了開發週期，提高了開發效率。選項 A 錯誤，其目標是減少程式碼。選項 C 錯誤，也適用於非專業人員。選項 D 錯誤，可以開發功能更複雜的應用程式。
#2
★★★★
關於 No Code / Low Code 平台，下列敘述何者較正確？
A
No Code 和 Low Code 完全相同
B
Low Code 平台不需要任何程式設計知識
C
Low Code 平台相較於 No Code 更適合開發需要較高靈活性和可擴展性的解決方案
D
No Code 平台可以無限客製化
答案解析
No Code 平台通常完全不需要編寫程式碼，適合沒有技術背景的使用者快速搭建簡單應用。而 Low Code 平台則允許使用者在視覺化開發的基礎上，透過少量編寫程式碼來實現更複雜的邏輯或客製化功能，因此提供了更高的靈活性和擴展性，適合開發需求較複雜的應用。選項 A 錯誤，兩者不同。選項 B 錯誤，Low Code 可能需要少量編碼。選項 D 錯誤，No Code 的客製化能力通常受限於平台提供的功能。
#3
★★★★
No Code/Low Code 平台最主要的優勢不包含下列何者？
A
加速應用程式的開發速度
B
降低開發的技術門檻，讓更多人參與開發
C
提供無限的客製化能力與最高的執行效能
D
降低應用程式的初期開發成本
答案解析
No Code/Low Code 平台的核心優勢在於速度、易用性和成本效益，可以快速開發、降低門檻、減少初期投入。然而，其主要限制在於客製化程度和效能。由於平台的抽象層和預設框架，對於非常特殊或複雜的功能，其客製化能力會受到限制。同時，相較於由專業工程師從頭編寫的優化程式碼，No Code/Low Code 平台生成的應用在執行效能或資源利用率上可能不是最高的，尤其是在處理高併發或大規模數據時。因此，選項 C 指出的「無限客製化」和「最高效能」並非其主要優勢，反而是可能的限制。
#4
★★★★
關於 No-Code AI 工具，下列敘述何者最為準確？
A
已經完全取代傳統的 AI 開發模式
B
只適用於大型企業
C
是一種降低 AI 技術複雜性和開發成本的新興方法
D
所有 No-Code AI 工具都具有完全相同的功能和性能
答案解析
No-Code AI 工具旨在讓沒有深厚程式設計或 AI 專業知識的使用者，也能透過圖形化介面或預配置模組來構建和部署 AI 應用。這種方法的主要目的是降低技術門檻，簡化開發流程，從而減少開發所需的時間和成本，使得 AI 技術更加普及化。它並未完全取代傳統開發模式（選項 A），因為傳統模式在客製化和效能優化上仍有優勢。它也不僅限於大型企業，中小型企業甚至個人都能使用（選項 B）。不同的 No-Code AI 工具有不同的功能側重和性能表現（選項 D）。
#5
★★★★
在選擇 No Code / Low Code 平台時，若主要考量是應用程式未來的功能擴充與和其他系統整合的能力，下列哪個因素最為重要？
A
平台的購買成本
B
平台的技術整合能力和自訂性
C
使用者的程式設計能力
D
平台的市場評價
答案解析
當考慮到應用程式未來的擴展性（增加新功能）和整合性（與現有或其他系統連接）時，平台本身的技術能力變得至關重要。這包括平台是否提供足夠的 API (Application Programming Interface) 或連接器來與外部系統互動，以及它允許多少程度的自訂開發（例如透過 Low Code 編寫特定邏輯）。高整合能力和自訂性意味著應用程式能更容易地適應未來的變化和需求。雖然成本（A）、使用者能力（C）和市場評價（D）也是考量因素，但在功能擴展和整合方面，技術整合能力和自訂性（B）是最直接相關且最重要的。
#6
★★★
No Code/Low Code 平台的主要目標使用者是誰？
A
僅限於資深的軟體工程師
B
僅限於數據科學家
C
包含業務人員、公民開發者以及專業開發人員
D
僅限於專案經理
答案解析
No Code/Low Code 的核心理念是開發民主化，讓更多不同技術背景的人都能參與應用程式的創建。No Code 平台主要面向沒有程式設計背景的業務人員或公民開發者，讓他們能自行解決簡單的自動化或應用需求。Low Code 平台則既能賦能公民開發者，也能提高專業開發人員的生產力，讓他們可以更快地構建應用程式骨架，並專注於編寫複雜的客製化部分。因此，其目標使用者範圍廣泛，並非僅限於特定技術角色。
#7
★★★
使用 No Code/Low Code 平台開發應用程式時，可能面臨的潛在限制或風險不包含下列何者？
A
對於高度客製化的功能可能支援不足
B
可能產生供應商鎖定 (Vendor Lock-in) 的風險
C
開發速度一定比傳統開發慢
D
對於極高效能或低延遲的需求可能難以滿足
答案解析
No Code/Low Code 平台的主要優勢之一就是加速開發速度，相較於從零開始的傳統開發模式，它通常能夠更快地交付應用程式。因此，選項 C 描述的情況與其優勢相反，是錯誤的。其他選項則是 No Code/Low Code 平台可能存在的限制：平台提供的功能和元件可能無法滿足所有客製化需求（A）；過度依賴特定平台可能導致未來難以遷移（B）；對於需要極致效能優化的應用可能表現不佳（D）。
#8
★★★★
關於生成式 AI 與 No Code / Low Code 平台的應用，下列何者最不適合？
A
利用平台自動生成應用程式的部分程式碼
B
在平台上自動化生成行銷文案或報告草稿
C
透過平台快速開發具有 AI 功能（如聊天機器人）的個人化 App
D
利用平台自動化生成具有法律效力的判決書
答案解析
No Code / Low Code 平台結合生成式 AI 可以實現許多應用，例如輔助程式碼生成、內容創作（文案、報告）、快速開發帶有 AI 功能的應用等（選項 A, B, C）。然而，法律判決具有極高的嚴肅性、專業性和權威性，需要嚴謹的法律推理和事實認定，並且直接影響當事人的權利義務。目前的 生成式 AI 在理解複雜法律情境、進行精確推理和確保判決公正性方面仍有侷限，且可能產生偏見或錯誤。因此，自動化生成具有法律效力的判決書（選項 D）是極不適合且具有高度風險的應用。
#9
★★★
企業導入 Low Code 平台以加速內部工具開發時，最需要關注的管理議題是什麼？
A
平台介面的美觀程度
B
是否能完全取代專業開發人員
C
如何確保公民開發者開發的應用程式符合品質、安全與合規標準
D
平台的程式碼生成效率
答案解析
當企業賦能公民開發者使用 Low Code 平台開發應用時，一個主要的管理挑戰是如何確保這些由非專業開發人員創建的應用程式的品質。這包括了應用程式的穩定性、安全性（是否遵循安全規範、是否存在漏洞）、數據隱私保護、以及是否符合企業和行業的合規要求。缺乏有效的治理和標準可能導致「影子 IT」(Shadow IT) 問題加劇，帶來營運和安全風險。因此，建立相應的治理框架、開發標準和審核流程至關重要。
#10
★★
下列哪種應用開發場景最不適合優先考慮使用 No Code 平台？
A
快速搭建一個產品原型 (Prototype) 以驗證市場想法
B
開發一個內部使用的簡單審批流程應用
C
製作一個活動報名的登陸頁面 (Landing Page)
D
開發一個需要處理每秒數百萬次請求的高併發交易系統
答案解析
No Code 平台非常適合快速開發、需求相對簡單、對性能要求不極端的場景，例如原型設計、內部工具、簡單網頁等（選項 A, B, C）。然而，對於需要極高效能、低延遲、高併發處理能力的核心交易系統或底層基礎設施（選項 D），No Code 平台通常難以滿足其嚴苛的性能和穩定性要求，這類系統通常需要經驗豐富的工程師使用傳統的程式設計語言進行深度優化和開發。
#11
★★★
相較於傳統軟體開發，No Code/Low Code 開發模式在哪方面可能較為不足？
A
開發速度
B
降低初期成本
C
對底層架構的控制力與程式碼優化的空間
D
讓非技術人員參與開發
答案解析
No Code/Low Code 平台為了實現快速開發和易用性，通常會封裝大量的底層細節，提供一個較高層次的抽象介面。這雖然簡化了開發，但也意味著開發者對於底層系統架構的控制能力會受到限制，並且難以進行細粒度的程式碼級別效能優化。傳統開發模式雖然速度較慢、成本較高，但提供了完全的控制權和最大的優化空間。速度、成本和賦能非技術人員是 No Code/Low Code 的優勢。
#12
★★★
"公民開發者"(Citizen Developer) 通常是指哪類人群？
A
擁有數十年經驗的資深軟體架構師
B
專業的 AI 演算法研究員
C
具備業務知識但程式設計技能有限或沒有，使用被認可的工具開發應用的員工
D
負責採購 No Code/Low Code 平台的 IT 部門主管
答案解析
公民開發者是指那些主要職責並非軟體開發，但利用企業批准或提供的 No Code/Low Code 工具，為自己或團隊創建應用程式以解決業務問題的員工。他們通常深入了解業務需求，但不具備傳統程式設計師的專業技能。No Code/Low Code 平台的普及使得這類角色越來越普遍。
#13
★★
以下哪項最不可能是 No Code/Low Code 平台提供的核心功能？
A
視覺化介面編輯器（Visual Interface Editor）
B
預建的應用程式模板（Pre-built Templates）
C
數據模型設計與管理（Data Modeling）
D
提供底層作業系統核心的修改權限
答案解析
No Code/Low Code 平台的核心在於抽象化和簡化開發過程。它們通常提供視覺化編輯器（A）、模板（B）和某種形式的數據管理功能（C）來幫助使用者快速構建應用。然而，為了確保平台的穩定性、安全性以及簡化開發體驗，它們幾乎不會提供對底層作業系統核心的直接修改權限（D）。這類底層操作屬於傳統系統級開發的範疇，與 No Code/Low Code 的高層次抽象理念背道而馳。
#14
★★★
使用 No Code/Low Code 平台開發的應用程式，其可擴展性（Scalability）主要取決於什麼？
A
開發者的程式設計能力
B
平台本身的架構設計和雲基礎設施
C
應用程式介面的美觀程度
D
使用的模板數量
答案解析
No Code/Low Code 應用程式的可擴展性（即處理更多用戶、更大數據量或更高負載的能力）主要受到其運行所在的平台基礎架構的限制。平台供應商如何設計其後端架構、使用的雲服務（如伺服器、數據庫）以及平台本身的資源管理機制，直接決定了在其上構建的應用程式能夠擴展到什麼程度。雖然開發者的設計（例如數據模型設計）也會有影響，但根本性的擴展能力是由平台決定的。
#15
★★★
在評估是否採用 No Code/Low Code 解決方案時，下列哪項不是重要的考量因素？
A
開發速度與上市時間 (Time-to-Market)
B
開發與維護成本
C
未來的功能擴展性與整合需求
D
平台創辦人的個人背景
答案解析
評估 No Code/Low Code 方案時，應關注其對業務目標和技術需求的匹配程度。重要的考量因素包括：能否快速交付以滿足市場需求（A）、總體擁有成本（包括開發、授權、維護）（B）、平台是否能支持未來的業務發展和系統整合（C）、平台的安全性、穩定性、供應商的支援能力、社群活躍度等。平台創辦人的個人背景（D）通常與平台的技術能力和適用性沒有直接關係，不是主要的評估依據。
#16
★★★★
No Code/Low Code 平台通常透過什麼方式來實現業務邏輯？
A
視覺化流程圖、預設規則引擎或少量腳本語言
B
只能透過編寫複雜的 Java 或 C++ 程式碼
C
完全依賴人工智慧自動生成所有邏輯
D
必須外聘專業顧問來定義所有業務規則
答案解析
No Code/Low Code 平台的核心是簡化邏輯的實現方式。No Code 平台通常提供視覺化的流程設計器（例如拖放節點來表示步驟和條件）或基於表單的規則配置，讓使用者可以直觀地定義業務邏輯。Low Code 平台在此基礎上，還可能允許使用者編寫少量的腳本（如 JavaScript）或表達式來處理更複雜的邏輯。它們的目標是盡量避免需要編寫大量傳統程式碼（B），雖然可能整合 AI，但不會完全依賴 AI 生成所有邏輯（C），也不一定需要外聘顧問（D）。
#17
★★★
當企業內部大量使用 No Code/Low Code 工具後，可能會出現應用程式數量激增、缺乏統一管理的情況，這種現象稱為？
A
技術債 (Technical Debt)
B
應用程式蔓延 (App Sprawl) 或 影子 IT (Shadow IT)
C
供應商鎖定 (Vendor Lock-in)
D
敏捷開發 (Agile Development)
答案解析
當公民開發者大量使用 No Code/Low Code 工具自行開發應用程式，而缺乏中央的 IT 部門治理和 oversight 時，容易導致應用程式數量失控、標準不一、重複開發、難以維護和整合，甚至可能引發安全和合規風險。這種由業務部門自行開發、未經 IT 部門正式批准或管理的系統被稱為「影子 IT」，而應用程式數量失控的現象則稱為「應用程式蔓延」。技術債是指為了短期速度而犧牲長期品質的選擇。供應商鎖定是依賴特定廠商的風險。敏捷開發是一種開發方法論。
#18
★★
下列何者是使用 No Code/Low Code 平台可以快速構建的應用範例？
A
開發一個全新的作業系統核心
B
設計一套用於大型粒子對撞機的模擬軟體
C
建立一個收集員工意見回饋的線上表單與自動通知流程
D
編寫一個控制精密工業機器人的即時控制系統
答案解析
No Code/Low Code 平台最擅長處理的是常見的業務流程自動化、數據收集與展示、簡單的內部管理工具等。建立一個收集意見回饋的表單，並在收到回饋後自動發送通知郵件，這是一個相對標準化的流程，可以利用平台提供的表單元件、工作流引擎和郵件通知功能快速搭建（選項 C）。而作業系統核心（A）、大型科學模擬軟體（B）、即時工業控制系統（D）都涉及底層硬體互動、複雜演算法和極高的性能/穩定性要求，遠超出一般 No Code/Low Code 平台的適用範圍。
#19
★★★
企業在選擇 No Code/Low Code 平台時，針對安全性的考量，以下何者最不重要？
A
平台是否提供角色基礎的存取控制 (Role-Based Access Control, RBAC)
B
平台的數據加密 (Encryption) 措施（傳輸中與靜止時）
C
平台供應商是否通過相關的安全認證（如 ISO 27001）
D
平台介面顏色主題的可自訂性
答案解析
評估 No Code/Low Code 平台的安全性時，應著重於其保護數據和應用程式的能力。重要的考量包括：是否有完善的權限管理機制以控制不同使用者對數據和功能的存取（A）；數據在傳輸和儲存時是否被加密以防止竊取（B）；供應商是否遵循業界的安全標準和最佳實踐，並獲得相關認證（C）；平台是否有漏洞管理和更新機制等。平台介面的顏色主題（D）屬於使用者體驗或外觀設計的範疇，與平台的安全性沒有直接關係。
#20
★★
相較於 No Code，Low Code 平台通常提供什麼額外的能力？
A
完全不需要任何技術知識
B
允許開發者編寫自訂程式碼來擴展功能或整合系統
C
開發速度一定比 No Code 慢
D
只能由專業程式設計師使用
答案解析
No Code 平台強調完全無需編碼，透過純視覺化操作完成應用開發。Low Code 平台的核心區別在於它在提供視覺化開發工具的同時，保留了編寫程式碼的選項。這使得開發者（包括專業開發者和具備一定技術能力的公民開發者）能夠在需要時，透過編寫少量自訂程式碼（例如用於複雜的業務邏輯、與特殊系統的整合、或特定UI元件的客製化）來突破平台的內建限制，實現更高的靈活性和功能擴展性。

★★★
